Playing around with autovectorization

March 26, 2020

It all started with a a message on the Houston Linux User Group (HLUG) discord:

If you’ve ever interacted with a linux users group, you’ll know that it didn’t take very long for the conversation to go from the politics of COVID-19 (booooring), to very quickly settling on a longer conversation about SIMD intrinsics.

inetknght was talking about needing a function that took a 16x4 (64-bit, not bytes) matrix and produced a 16x8 (128-bit) matrix where the 4 bits in the original matrix were zero-extended and distributed in a certain way based on parity, as a part of a larger SIMD pipeline.

Some of us had some doubt that operating on 4-bit datum would be something that vectorization would be able to handle gracefully. Unbeknownst to us, inetknght had poured through the Intel Intrinsics Guide and already figured this out, and in heroic systems programmer fashion, he started screen-sharing on Discord, popped open Godbolt, and got to work.

The thing we’re trying to do, in Python

Before jumping into the C++, I thought it might be simpler to explain the operation with python and a 4-bool tuple type. That was a mistake, but I wrote the code anyway. You can see it all here. Assuming you can figure out what the functions do from their names, here’s a relevant snippet:

def convert_one_row(i: row_in) -> row_out:
    """ Zero pad a 4-bit tuple representation to an 8-bit tuple representation """
    # i is already a "row", so the MSB is the lower index.
    return (False, False, False, False, i[3], i[2], i[1], i[0])

def convert_mats(m_in: mat_in) -> mat_out:
    """ Given a 16x4 tuple representation, produce a 16x8 tuple with zero-extension """
    return (
        convert_one_row(m_in[14]),
        convert_one_row(m_in[12]),
        convert_one_row(m_in[10]),
        convert_one_row(m_in[8]),
        convert_one_row(m_in[6]),
        convert_one_row(m_in[4]),
        convert_one_row(m_in[2]),
        convert_one_row(m_in[0]),
        convert_one_row(m_in[15]),
        convert_one_row(m_in[13]),
        convert_one_row(m_in[11]),
        convert_one_row(m_in[9]),
        convert_one_row(m_in[7]),
        convert_one_row(m_in[5]),
        convert_one_row(m_in[3]),
        convert_one_row(m_in[1]),
    )

Don’t pay any attention to how the bits are actually laid out in memory here, I’m just trying to abstractly show the transformation we’re trying to accomplish.

With Intrinsics

The intrinsics-based C++ to do the same is as follows:

#include <cstdint>
#include <emmintrin.h>

__m128i unpack(const uint64_t& s)
{
    __m128i ab = _mm_set_epi64x(s >> 4, s);
    ab &= _mm_set1_epi8(0x0F);
    return ab;
}

This produces the following assembly:

.LCPI0_0:
        .zero   16,15
unpack(unsigned long const&):       # @unpack(unsigned long const&)
        mov     rax, qword ptr [rdi]
        vmovq   xmm0, rax
        shr     rax, 4
        vmovq   xmm1, rax
        vpunpcklqdq     xmm0, xmm0, xmm1 # xmm0 = xmm0[0],xmm1[0]
        vpand   xmm0, xmm0, xmmword ptr [rip + .LCPI0_0]
        ret

Without intrinsics?

That code is gobbledygook to the average C programmer. Low-level code that deals with bit manipulation is always going to be a little harder to grok, but wouldn’t it be nice if we could just write straight-up C and get what we want?

After playing with it for some time, it turns out that you can.

#include <cstdint>
typedef long long __m128i __attribute__ ((__vector_size__ (16), __may_alias__));

__m128i unpack_no_intrinsics(const uint64_t& s)
{
        __m128i to_ret = {0,0};
        for (auto shift = 0; shift < 64; shift += 8) {
                uint8_t byte = (uint8_t)(s >> shift);
                uint64_t even  = (byte & 0x0F);
                uint64_t odd = ((byte & 0xF0) >> 4);
                to_ret[0] |= even << shift;
                to_ret[1] |= odd  << shift;
        }
        return to_ret;
}

This produces the same exact code:

.LCPI0_0:
  .zero 16,15
unpack_no_intrinsics(unsigned long const&): # @unpack_no_intrinsics(unsigned long const&)
  mov rax, qword ptr [rdi]
  vmovq xmm0, rax
  shr rax, 4
  vmovq xmm1, rax
  vpunpcklqdq xmm0, xmm0, xmm1 # xmm0 = xmm0[0],xmm1[0]
  vpand xmm0, xmm0, xmmword ptr [rip + .LCPI0_0]
  ret

Conclusions

  • Your local linux user group is an awesome place to find interesting conversation. Go get involved with yours!
  • I wish that MyPy/typing had better syntax for fixed-size homogeneous tuples without writing Tuple[Type, Type, Type, Type, ...]. It also (in my experience) can’t unwrap loops to reason about tuple lengths. The gist above does pass MyPy, but it took some refactoring and unrolling.
  • Generally speaking, it is possible to write straight-up C/C++ and get fast code. Some advice to my future self:
    • You have to look at the assembly coming out of your compiler.
    • Try a build with -O3 and see what happens.
    • Try a build with a newer version of your compiler and see what happens.
    • Try a build with a different compiler and see what happens.
    • All of this requires sane types with the right attributes (ie: alignment). If you think your compiler is generating inefficient code, it’s probably worth your time to try adding the attributes to your existing types before you go all-in and rewrite it to use intrinsics.
    • If there’s a loop that you suspect is capable of being autovectorized, add pragmas that tell your compiler you definitely want it to be vectorized, and add the corresponding compiler flags that emits warnings when that doesn’t happen. Autovectorization lends itself to regressions and it’s important that you are able to notice when that happens. One big benefit to using intrinics is that you don’t have to worry about that sort of regression.
  • How this plays in a larger SIMD pipeline is another question entirely.
